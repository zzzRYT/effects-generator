# PRD — song-index (곡 목록 / 검색 진입점)

- **Feature slug**: song-index
- **Brainstorm/설계**: docs/plans/2026-06-21-song-index-design.md
- **상태**: DRAFT

## 무엇 (What)
홈(`/`)을 곡 라이브러리의 **진입점**으로 만든다. 모든 곡을 정적 목록으로 보여주고, 검색창 1개 +
rig 필터 칩으로 즉시 좁힌다. 검색어·활성 rig는 URL(`?q=&rig=`)에 실려 공유·북마크 가능하다.
현재 홈(#1 임시 정적 링크 목록)을 이 본격 진입점으로 교체한다.
(genre 는 짧은 태그가 아니라 긴 서술문이라 칩 대신 **검색 대상**에 포함 — 키워드 검색으로 커버.)

## 왜 (Why) / 누구를 위해
지금 홈은 전체 곡을 단순 나열만 한다. 곡이 늘면(tone-builder 가 계속 추가) "그 곡 어딨지"를 빠르게
찾기 어렵다. GP-150 유저(나·밴드)가 **아티스트/곡명으로 검색하거나 rig·장르로 걸러** 원하는 패치로
바로 들어가고, 특정 필터 뷰(`?q=oasis`)를 멤버에게 공유할 수 있어야 한다. 진입점이 제대로 서야 #1·#2
의 가치(패치·변주)가 발견 가능해진다.

## 수용 기준 (측정 가능 — 테스트로 검증)
- [ ] AC1: 모든 곡이 정적 목록으로 렌더된다(행마다 아티스트·제목·rig·변주수, 곡 상세 링크). (vitest + 빌드 정적)
- [ ] AC2: 검색창에 입력하면 artist/title/genre 부분문자열(대소문자 무시) 매칭 행만 보이고 나머지는 숨겨진다. (Playwright)
- [ ] AC3: rig 칩을 고르면 그 rig 곡만. rig 칩은 단일 선택(전체 또는 하나). 검색어·rig는 AND 결합. (Playwright)
- [ ] AC4: 활성 검색어·rig가 URL `?q=&rig=`에 반영되고, 그 URL로 딥링크 진입 시 같은 필터 상태로 열린다. (vitest parseFilters + Playwright)
- [ ] AC5: 필터 결과가 0곡이면 "검색 결과 없음" 메시지 + 필터 초기화 링크가 표시되고 링크가 작동한다(crash 0). (Playwright edge-3.6)
- [ ] AC6: 활성 필터 칩은 `aria-pressed` + 시각 강조(색만으로 구분 금지)로 표시된다. (vitest + Playwright)
- [ ] AC7: 결과 수가 `aria-live="polite"` 영역에 "N곡"으로 안내된다(필터 시 갱신). (Playwright)
- [ ] AC8: **no-JS에서 모든 곡 목록이 보인다**(필터 컨트롤은 미렌더 — 작동 안 하는 UI 숨김). (Playwright `javaScriptEnabled:false`)
- [ ] AC9: 검색창·칩이 키보드로 순회 가능하고 focus ring이 보인다. axe 위반 0. (Playwright)
- [ ] AC10: 320/375/768/1024/1440에서 오버플로 0, 터치 타깃 ≥44px, 칩바 wrap. (Playwright)
- [ ] AC11: 빌드가 정적(SSG/Static) 유지 — 홈 `○ Static`, 런타임 페치 0. 필터 아일랜드는 Suspense 격리. (next build 로그)

## 비목표 (Non-goals)
- 정렬(아티스트/제목순) → 미채택(소규모, 복잡도만 추가). 필요 시 후속.
- 다중 선택 칩(여러 rig/genre OR) → 미채택. 카테고리당 단일.
- 퍼지/오타 허용 검색, 검색 하이라이트 → 부분문자열로 충분(YAGNI).
- 무한스크롤/페이지네이션 → 정적 소규모 목록엔 불필요.
- 곡 상세/변주 뷰 → #1·#2. 제보 폼 → #4.
- 서버 사이드 필터 → 불가(정적, 런타임 서버 없음). 클라이언트 필터만.

## 열린 질문
- 행에 genre 표기를 추가할지(현재 rig·변주수만) — TRD에서 최소 형태로 결정.
- 검색 입력 디바운스 필요 여부 — 소규모라 불필요로 보되 TRD에서 확정.
